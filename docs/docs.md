# How cvmfs-debug-rescue Works

## Background

FUSE filesystems operate through a kernel-to-userspace protocol. When a
process performs a filesystem operation (like `stat()` or `read()`), the
kernel translates it into a FUSE request, sends it to the userspace daemon
via `/dev/fuse`, and puts the calling process into uninterruptible sleep
(D-state) while waiting for the daemon's response.

If the daemon never responds — due to a deadlock, network hang, or bug —
the calling process remains in D-state permanently. The kernel uses
uninterruptible sleep to protect filesystem consistency: allowing the process
to be interrupted mid-operation could leave kernel data structures in an
inconsistent state.

Standard debugging tools fail because they all depend on signal delivery:

| Tool | Mechanism | Why it fails on D-state |
|------|-----------|------------------------|
| `gdb -p` | `ptrace(PTRACE_ATTACH)` → sends `SIGSTOP` | Signal not delivered |
| `gcore` | Uses ptrace internally | Same as gdb |
| `kill -ABRT` | Sends `SIGABRT` to trigger core dump | Signal queued but never delivered |
| `kill -9` | Sends `SIGKILL` | Signal queued but never delivered |

## The Key Insight

The kernel maintains complete process state in `/proc/<pid>/` regardless of
scheduler state. These are direct reads from kernel data structures — they do
not send signals and do not require the process to be runnable.

This was verified experimentally: every tested `/proc/<pid>/` file remains
fully readable for D-state processes, including `/proc/<pid>/mem` which
provides raw access to the process's entire virtual address space.

## Development Process

### Iteration 1: Bare Minimum Core

The first version extracted only RIP and RSP from `/proc/<pid>/syscall` and
dumped all readable memory regions into an ELF core file with a single
`NT_PRSTATUS` note.

**Result:** GDB loaded the core and found the stack, but showed `?? ()` for
every frame. The stack unwinder walked 30 frames deep — the memory was there
— but GDB could not map addresses to symbols because of ASLR. Every binary
and shared library loads at a random address, and our core had no metadata
telling GDB which files were loaded where.

The raw stack data was verified with `x/16xg $rsp`, confirming the memory
dump was valid.

### Iteration 2: Adding NT_FILE (File Mappings)

Added an `NT_FILE` note parsed from `/proc/<pid>/maps`, mapping memory
regions to their backing files (e.g., "addresses `0x7f23b498b000` to
`0x7f23b49af000` belong to `/usr/lib/libc.so.6`").

**Result:** GDB still showed `?? ()` by default. However, manual symbol
loading with `add-symbol-file` now worked:


```

(gdb) add-symbol-file /usr/bin/stat 0x56473e6c3000
(gdb) bt
#0  0x00007f481b7d179e in ?? ()
#1  0x000056473e6c4743 in statx_timestamp_to_timespec (tsx=...)
at src/statx.h:28
#2  statx_to_stat (...) at src/statx.h:47
#3  do_stat (...) at src/stat.c:1409

```

First real function names and source lines from a D-state process.

The reason GDB did not auto-resolve symbols: without the auxiliary vector
(`AT_PHDR`, `AT_ENTRY`, etc.), GDB cannot locate the dynamic linker's
internal `link_map` structure that chains all loaded shared libraries
together.

### Iteration 3: Adding NT_AUXV (Auxiliary Vector)

Added an `NT_AUXV` note by reading `/proc/<pid>/auxv` — a binary blob the
kernel exposes directly. This provides the information GDB needs to find the
dynamic linker and automatically resolve all shared library symbols.

Also added `NT_PRPSINFO` (process name and state) and improved register
extraction to capture all available registers from `/proc/<pid>/syscall`:
RDI, RSI, RDX, R10, R8, R9, RSP, and RIP (8 registers total instead of 2).

**Result:**


```

Core was generated by `/usr/bin/stat /tmp/testmount/hang'.

#0  statx (fd=-100, path=0x7ffe1fecdc35 "/tmp/testmount/hang",
flags=2304, mask=4095, buf=0x7ffe1fecd310)
at ../sysdeps/unix/sysv/linux/statx.c:28
#1  0x000055947edf8743 in do_stat (...) at src/stat.c:1361

```

GDB auto-downloaded debug symbols via debuginfod, identified the process
from `NT_PRPSINFO`, resolved all library addresses through `NT_AUXV`, and
displayed the exact source line with function arguments.

## Register Mapping

`/proc/<pid>/syscall` format on x86_64:


```

syscall_nr  arg0  arg1  arg2  arg3  arg4  arg5  RSP  RIP

```

The syscall arguments correspond to registers per the x86_64 calling
convention:

| Field | Register | Index in user_regs_struct |
|-------|----------|--------------------------|
| arg0 | RDI | 14 |
| arg1 | RSI | 13 |
| arg2 | RDX | 12 |
| arg3 | R10 | 7 |
| arg4 | R8 | 9 |
| arg5 | R9 | 8 |
| RSP | RSP | 19 |
| RIP | RIP | 16 |
| syscall_nr | RAX / ORIG_RAX | 10, 15 |

Registers not available through procfs: RBP, RBX, RCX, R11, R12, R13, R14,
R15, EFLAGS. These are saved in the kernel's `pt_regs` structure on the
kernel stack but not exported to userspace.

## ELF Core File Structure

The generated core file contains:


```

┌────────────────────────────┐
│ ELF Header (ET_CORE)       │
├────────────────────────────┤
│ Program Header: PT_NOTE    │
│ Program Header: PT_LOAD ×N │
├────────────────────────────┤
│ Note: NT_PRSTATUS          │  Register state (per thread)
│ Note: NT_PRPSINFO          │  Process name, PID, state
│ Note: NT_FILE              │  Memory-to-file mappings
│ Note: NT_AUXV              │  Dynamic linker info
├────────────────────────────┤
│ Memory Segment 1           │  /usr/bin/stat .text
│ Memory Segment 2           │  /usr/lib/libc.so.6
│ ...                        │
│ Memory Segment N           │  [stack]
└────────────────────────────┘

```

Each `PT_LOAD` segment corresponds to a readable memory region from
`/proc/<pid>/maps`. The `NT_FILE` note maps these segments back to their
backing files so GDB can locate debug symbols.

## Multi-Thread Support

For multi-threaded processes (like FUSE daemons), the tool:

1. Enumerates threads via `/proc/<pid>/task/`
2. Reads `/proc/<pid>/task/<tid>/syscall` for each thread's register state
3. Writes one `NT_PRSTATUS` note per thread
4. GDB shows all threads with `info threads` and `thread apply all bt`

## Known Issue: Thread Stack Guard Pages

Thread stacks allocated by libpthread reside in large anonymous memory
regions (~8MB per thread) that include a guard page (`PROT_NONE`) at the
low end. When reading these regions sequentially through `/proc/<pid>/mem`,
the guard page causes a read failure that can affect subsequent reads within
the same region, resulting in zeroed stack data.

This does not affect the main thread (whose stack is in a separate `[stack]`
region) or single-threaded processes. A fix (per-chunk seeking with
independent `lseek` calls for each page) has been implemented to survive guard pages.
